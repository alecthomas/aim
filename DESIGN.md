# migraitor — Design Document

User maintains a single `schema.sql`. migraitor uses an LLM to generate migration statements, then deterministically verifies them by comparing two ephemeral databases. If they don't match, it retries with the plain text diff as feedback.

## How It Works

1. **Derive previous state.** Replay `migrations/001.sql … N.sql` (up files) against an ephemeral DB.
2. **Generate.** The LLM agent produces candidate up and down migrations. It has tools to read the current `schema.sql`, list/read existing migrations, and inspect the SQL dialect. It returns both up and down SQL.
3. **Verify up.** Stand up two ephemeral databases:
   - **DB-Left:** run `schema.sql` directly (desired end state).
   - **DB-Right:** replay prior migrations, then apply the candidate up migration.
   - Compare using the engine's `diff` method. If the output is empty, the up migration is correct.
4. **Verify down.** Apply the candidate down migration to DB-Right. Diff it against the previous state (from step 1). If empty, the down migration is correct.
5. **Retry.** If either diff is non-empty, feed it back to the LLM as plain text and repeat from step 2, up to `max_retries`.
6. **Persist.** Write both files to `migrations/`.

## CLI

```
migraitor init      # create config + schema.sql
migraitor plan      # generate + verify, print result
migraitor apply     # generate + verify + apply to target DB
migraitor rollback  # apply the most recent down migration
```

Key flags: `--engine postgres|mysql|sqlite`, `--connection <URL>`, `--max-retries <N>`, `--model <MODEL>`, `--provider <PROVIDER>`.

## Key Abstractions

### `DatabaseEngine` trait

Portability boundary. Every supported database implements it.

```rust
#[async_trait]
pub trait DatabaseEngine: Send + Sync {
    async fn create_ephemeral(&self) -> Result<EphemeralDb>;
    async fn execute(&self, db: &EphemeralDb, sql: &str) -> Result<()>;
    async fn diff(&self, a: &EphemeralDb, b: &EphemeralDb) -> Result<String>;
    async fn drop_ephemeral(&self, db: EphemeralDb) -> Result<()>;
    fn dialect_description(&self) -> &str;
}
```

`diff` returns an empty string on match, or a plain text description of the differences. Each engine implements this however makes sense — querying `information_schema`, `sqlite_master`, etc.

Ephemeral databases are Docker containers. Each `create_ephemeral` call spins up a container with the appropriate image (`postgres`, `mysql`, `sqlite3`), `drop_ephemeral` removes it. Docker is the only external dependency.

### Agent Tools

The LLM agent is given the following tools via rig:

- **`read_schema`** — returns the contents of `schema.sql`.
- **`list_migrations`** — returns the ordered list of existing migration filenames.
- **`read_migration`** — returns the contents of a specific migration file by name.

The agent uses these to understand the current state before producing migration SQL. The orchestrator provides the dialect and previous DDL in the system prompt; the tools let the agent pull additional context as needed.

## LLM Integration

Uses [rig](https://github.com/0xPlaygrounds/rig) for provider-agnostic LLM access. The system prompt includes the dialect and previous DDL. The agent has tools to read the schema and migrations. It produces both up and down SQL in a single response. On retry, the plain text diff is appended as a correction message.

## Migration Files

```
migrations/
├── 001_initial.up.sql
├── 001_initial.down.sql
├── 002_add_email_to_users.up.sql
├── 002_add_email_to_users.down.sql
├── 003_create_orders_table.up.sql
└── 003_create_orders_table.down.sql
```

Each file has a header with timestamp, engine, and SHA-256 checksum. A `migraitor_history` table in the target DB tracks what's been applied. The description is generated by a secondary LLM call.

## Security

LLM output is untrusted SQL. It is only ever executed against ephemeral Docker containers during verification — never against the target until verification passes. Connection strings are never sent to the LLM.
